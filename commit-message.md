# Commit 訊息怎麼寫

## 為什麼不要亂寫 update

`git log` 是專案的日記。等你過幾天回頭看，或是想找「這個檔案是什麼時候、為什麼被改的」，commit message 就是你唯一的線索。

如果訊息都是這樣：

```
update
fix
123
改了一下
```

那這份日記等於沒寫，半年後沒人知道（包括你自己）當時為什麼改這行。

只要養成習慣，用一句話講清楚「做了什麼」，之後找東西、回溯問題都會輕鬆很多。

---

## 推薦格式

```
<type>: <做了什麼>
```

例如：

```bash
git commit -m "docs: 新增 commit 訊息教學"
git commit -m "fix: 修正登入按鈕沒反應的問題"
```

`type` 是這次改動的「類別」，`:` 後面用一句話講清楚內容就好。中文寫 commit message 完全沒問題，看得懂、好維護比什麼都重要。

---

## 常見 type

新手先記這四個就夠用：

| type | 用途 | 範例 |
|---|---|---|
| `feat` | 新增功能 | `feat: 新增登入頁面` |
| `fix` | 修 bug | `fix: 修正登出後頁面沒跳轉的問題` |
| `docs` | 只改文件，不動程式邏輯 | `docs: 補上 README 安裝步驟` |
| `chore` | 雜項（調設定、改檔名等，跟功能無關） | `chore: 調整 .gitignore` |

（之後熟悉了，還會看到 `style`、`refactor`、`test` 等其他 type，但現階段不用急著背。）

---

## 好與不好的範例

❌ 不好：
```
update
fix bug
改了一些東西
```

✅ 好：
```
feat: 新增購物車刪除商品功能
fix: 修正手機版選單打不開的問題
docs: 更新安裝教學
chore: 移除沒用到的測試檔案
```

差別在哪？好的範例讓你不用打開 diff，光看訊息就知道這次 commit 做了什麼。

---

## 一個 commit 只做一件事

如果你發現訊息要用「然後」、「順便」、「還有」才講得完，通常代表這個 commit 做了太多事，該拆開。

❌ `fix: 修正登入問題，順便改了一下 README`

✅ 拆成兩個：
```
fix: 修正登入失敗的問題
docs: 更新 README
```

這樣以後如果某個改動出問題，可以單獨 revert 掉，不會牽連到其他不相關的內容。

---

## 小結

- 不要寫 `update`、`fix`、`123` 這種看不出內容的訊息
- 格式：`type: 做了什麼`
- 先記 `feat` / `fix` / `docs` / `chore` 這四個就夠
- 中文 commit message 沒問題
- 一個 commit 只做一件事

把這幾條養成習慣，之後再慢慢認識更進階的規範（像 scope、body、自動化工具）會更輕鬆，不用一次學完。
